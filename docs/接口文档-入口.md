# Apache Answer 接口文档入口

本文是接口文档入口，用来说明文档范围、基础约定、认证方式、访问控制分层和维护注意事项。

接口文档建议按下面顺序查看：

- 总体接口与权限分层：本文件
- 每个接口的请求参数、请求体模型、成功响应模型： [接口文档.md](接口文档.md)
- 原始 Swagger 产物： [swagger.yaml](./swagger.yaml) 和 [swagger.json](./swagger.json)

## 文档范围

- 服务健康检查、安装流程、公共 API、登录后 API、管理员 API
- 插件扩展接口，包括 connector、user-center、captcha、embed、render
- 每个接口的请求参数、请求体模型、成功响应模型

## 基础路径

- 公开和普通用户接口前缀：`/answer/api/v1`
- 管理后台接口前缀：`/answer/admin/api`
- 安装接口前缀：`/installation`
- Swagger UI：`/swagger/*any`
- 健康检查：`/healthz`

## 认证方式

使用 `Authorization` 请求头传递访问令牌。服务端会自动去掉 `Bearer ` 前缀，因此以下两种写法都可用：

```http
Authorization: Bearer <access_token>
Authorization: <access_token>
```

若请求头没有 `Authorization`，部分接口也支持通过查询参数 `Authorization` 传递 token。

## 访问控制分层

当前服务不是简单地按“公开 / 登录 / 管理员”三层划分，而是分成以下几类：

| 访问级别 | 说明 |
| --- | --- |
| Public | 完全公开，无需登录 |
| Public + token | 完全公开；如果携带 token，可返回当前用户相关数据 |
| Optional Auth | 可匿名访问；如果站点启用了私有访问模式，则必须登录且邮箱已验证 |
| Auth (any status) | 必须登录，但不强制要求账号已激活 |
| Auth | 必须登录，且邮箱已验证、账号可用、未被封禁 |
| Admin | 必须是管理员 |

## 公共响应结构

绝大多数 JSON API 都使用统一响应包裹：

```json
{
  "code": 200,
  "reason": "success",
  "msg": "Success",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | `int` | HTTP 状态码 |
| `reason` | `string` | 业务原因码 |
| `msg` | `string` | 当前语言下的可读消息 |
| `data` | `any` | 业务数据，可能为对象、数组、分页对象或 `null` |

列表接口经常返回分页模型：

```json
{
  "count": 123,
  "list": []
}
```

## 常见状态码

| 状态码 | 含义 |
| --- | --- |
| `200` | 成功 |
| `400` | 请求参数错误 |
| `401` | 未登录或 token 无效 |
| `403` | 无权限 |
| `404` | 资源不存在 |
| `500` | 服务端错误 |

## 语言与跨域

- 语言通常通过 `Accept-Language`、Cookie 或用户配置决定
- 服务端统一启用了 CORS 中间件，允许浏览器端跨域调用

## Swagger 差异与维护说明

以下内容来自旧版接口文档核对结果，并已合并到当前文档体系：

- `GET /answer/api/v1/user/notification/config`：Swagger 注解曾写成 `POST`，但实际注册路由为 `GET`
- `/personal/question/page`：Swagger 中出现过无 `/answer/api/v1` 前缀的形式，实际生效路由是 `/answer/api/v1/personal/question/page`
- 部分插件或扩展类接口只有路由注册，没有 Swagger 注解，因此仅依赖 `swagger.json` 会漏文档
- `captcha/config`、`communication`、`question/partition`、`tag/secondary`、`topic`、`topic/interest`、`user/profession`、`connector/*`、`user-center/*` 等代码已生效接口，已在 [接口文档.md](接口文档.md) 中以“Swagger 未覆盖但代码已生效的接口”补充

## 推荐查看方式

如果你要对接接口：

1. 先看 [接口文档.md](接口文档.md) 定位接口
2. 再看对应的请求体/响应体模型定义
3. 如果和 Swagger 有出入，以代码实际注册路由为准

推荐继续维护官方 Swagger 时，先修正上述不一致项并补全遗漏控制器注解，再重新生成 `swagger.yaml` 与 `swagger.json`。
