# Apache Answer 接口文档

本文档基于当前仓库代码整理，主要以实际注册到 Gin 的路由为准，辅助参考现有的 [swagger.yaml](../../../../百度云同步空间/Code/answer/docs/swagger.yaml) 和 [swagger.json](../../../../百度云同步空间/Code/answer/docs/swagger.json)。相比现有 Swagger，这份文档额外补充了权限分层、插件路由、安装流程、以及当前代码里已生效但 Swagger 未完整覆盖的接口。

## 1. 文档范围

- 服务健康检查、安装流程、公共 API、登录后 API、管理员 API
- 插件扩展接口，包括 connector、user-center、captcha、embed、render
- 核心请求体字段说明
- 代码与现有 Swagger 的差异说明

## 2. 基础约定

### 2.1 基础路径

- 公开和普通用户接口前缀：`/answer/api/v1`
- 管理后台接口前缀：`/answer/admin/api`
- 安装接口前缀：`/installation`
- Swagger UI：`/swagger/*any`
- 健康检查：`/healthz`

### 2.2 认证方式

- 使用 `Authorization` 请求头传递访问令牌
- 服务端会自动去掉 `Bearer ` 前缀，因此以下两种写法都可用：

```http
Authorization: Bearer <access_token>
Authorization: <access_token>
```

- 若请求头没有 `Authorization`，部分接口也支持通过查询参数 `Authorization` 传递 token

### 2.3 多层访问控制

当前服务不是简单地按“公开/登录/管理员”三层划分，而是分成以下几类：

| 访问级别 | 说明 |
| --- | --- |
| Public | 完全公开，无需登录 |
| Public + token | 完全公开；如果携带 token，可返回当前用户相关数据 |
| Optional Auth | 可匿名访问；如果站点启用了私有访问模式，则必须登录且邮箱已验证 |
| Auth (any status) | 必须登录，但不强制要求账号已激活 |
| Auth | 必须登录，且邮箱已验证、账号可用、未被封禁 |
| Admin | 必须是管理员 |

### 2.4 公共响应结构

绝大多数 JSON API 都使用统一响应包裹：

```json
{
  "code": 200,
  "reason": "success",
  "msg": "Success",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | `int` | HTTP 状态码 |
| `reason` | `string` | 业务原因码 |
| `msg` | `string` | 当前语言下的可读消息 |
| `data` | `any` | 业务数据，可能为对象、数组、分页对象或 `null` |

### 2.5 分页返回结构

列表接口经常返回分页模型：

```json
{
  "count": 123,
  "list": []
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `count` | `int` | 总记录数 |
| `list` | `array` | 当前页数据 |

### 2.6 常见状态码

| 状态码 | 场景 |
| --- | --- |
| `200` | 请求成功 |
| `400` | 请求参数格式错误、校验失败 |
| `401` | 未登录、token 无效、账号被删除 |
| `403` | 权限不足、邮箱未验证、账号已封禁、站点限制访问 |
| `500` | 服务端未知错误 |

### 2.7 语言与跨域

- 服务端启用了 CORS，允许 `GET/POST/PUT/DELETE/OPTIONS`
- 可通过 `Accept-Language` 影响 `msg` 的翻译文本
- 静态资源和页面渲染还会走额外的请求头处理中间件

## 10. 与现有 Swagger 的差异

以下内容是本次整理时发现的实际情况：

### 10.1 当前代码已注册，但 Swagger 未完整覆盖的接口

- `/answer/api/v1/captcha/config`
- `/answer/api/v1/communication`
- `/answer/api/v1/question/partition`
- `/answer/api/v1/question/recommend/partition`
- `/answer/api/v1/questions/change_date_view`
- `/answer/api/v1/tag/secondary`
- `/answer/api/v1/topic`
- `/answer/api/v1/topic/interest`
- `/answer/api/v1/topic/interest/cancel`
- `/answer/api/v1/user/profession`
- `/answer/api/v1/user-center/*`
- `/answer/api/v1/connector/login/:name`
- `/answer/api/v1/connector/redirect/:name`

### 10.2 注解与真实路由存在偏差的点

- `GetUserNotificationConfig` 的 Swagger 注解写成了 `POST /answer/api/v1/user/notification/config`，但实际注册的是 `GET /answer/api/v1/user/notification/config`
- `personal question page` 在 Swagger 中出现为 `/personal/question/page`，但实际生效路由是 `/answer/api/v1/personal/question/page`
- 部分插件或扩展类接口只有路由注册，没有 Swagger 注解，因此仅依赖 `swagger.json` 会漏文档

### 10.3 使用建议

- 如果要继续维护官方 Swagger，建议优先修正上述不一致项
- 如果要给前端或第三方集成使用，优先以本文档和 `internal/router/*.go` 的实际注册逻辑为准
- 如果要自动生成 OpenAPI，建议先补全遗漏控制器上的注解，再重新执行 `script/gen-api.sh`
