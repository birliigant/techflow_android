当前 UI 形态：

- 单 `Activity`：`MainActivity`
- 多 Compose Screen：首页、详情、提问、独立登录、独立注册、我的、设置、搜索、标签、用户主页等
- 首页使用 Material3 PullToRefresh 下拉刷新问题流
- 我的未登录态使用简洁空状态，只保留登录 / 注册入口
- 首次启动会统一检查通知、图片访问等可运行时授权的系统权限；网络权限属于 Manifest 普通权限，Android 不支持运行时弹窗
- 登录和注册密码框使用 `PasswordVisualTransformation` 默认隐藏，并提供可视切换按钮
- 登录 / 注册成功使用带缓冲的 `SharedFlow` 一次性事件驱动 Toast 与自动跳转；登录成功会先用应用级 Context 展示 Toast，再清理认证路由栈并进入“我的”，避免状态恢复导致导航丢失
- 底部导航使用自定义紧凑 Compose 组件，保留系统手势安全区但降低导航内容区高度
- 顶层由 `TechFlowApp` 统一组织导航与页面装配

## 七、网络层

| 技术 | 版本 | 用途 |
| --- | --- | --- |
| Retrofit | 2.11.0 | 定义后端接口 |
| Gson Converter | 2.11.0 | JSON 序列化 / 反序列化 |
| OkHttp | 4.12.0 | HTTP 客户端 |
| OkHttp Logging Interceptor | 4.12.0 | 请求日志输出 |
| Gson | 跟随依赖树 | DTO 与本地会话 JSON 处理 |

网络层特点：

- 默认后端地址固定为 `https://answer.sipc115.com/`
- 通过 `ApiClientProvider` 按 `baseUrl + token` 生成并缓存 Retrofit 实例
- 自动注入与 Web 端一致的 `Authorization: <token>` 裸 token 请求头
- 点赞操作会先调用 `/answer/api/v1/permission` 进行权限预检，兼容按 action 分组的权限响应，并解析 403 错误体展示后端提示
- 详情页操作反馈通过 Android `Toast` 展示，避免滚动内容顶部提示不可见
- 提问接口按 Web 端实际行为提交 `title/content/partition/tags`，细分类标签可选，未填写时自动兜底到当前分区的“其他”标签
- 提问页标签选择按分区提供 Web 端风格的常用标签按钮，并支持加号打开“创建新标签”对话框，将自定义标签加入本次发帖请求
- 提问提交进入网络请求前会立即锁定提交按钮；创建问题接口使用短超时，并在超时后通过 Toast 进入待确认状态，再短轮询最新问题列表与搜索结果，兜底识别“后端已创建但客户端未收到成功响应”的灰色状态
- 删除整帖成功后会回到首页并重建首页路由，确保问题流重新拉取最新数据
- 详情页帖子和回答操作栏支持横向滚动，避免点赞、收藏、评论、分享、删除、举报等操作在小屏下被裁切
- 首页工具菜单、顶部用户菜单和我的页更多菜单均通过 Compose 弹出菜单锚定触发按钮，并使用统一下方间距
- Android 系统权限弹窗文案由系统语言决定，App 在唤起系统弹窗前提供中文说明弹窗
- 当前接口依据：
  - `docs/接口文档-入口.md`
  - `docs/接口文档.md`
  - `docs/swagger.json`
  - `docs/swagger.yaml`

已接入的核心接口能力包括：

- 站点信息
- 问题列表 / 问题详情
- 回答列表 / 发布回答 / 删除回答
- 评论列表 / 发布评论 / 回复评论
- 点赞 / 收藏 / 删除帖子 / 举报
- 登录 / 注册 / 获取当前用户
- 用户主页 / 收藏 / 声望 / 评论 / 得票 / 徽章
- 搜索
- 文件上传（发帖图片）

## 十二、工程特点总结

### 1. 技术选型偏现代 Android 原生

当前主栈已经稳定落在：

- Kotlin
- Jetpack Compose
- Navigation Compose
- Retrofit + OkHttp
- Room
- MMKV
- Coroutines

这使项目在 Android 原生方向上是清晰且统一的，没有混用多套 UI 或多套路由框架。

### 2. 结构轻量，但边界清晰

项目没有引入 Hilt/Koin 这类重量级 DI，而是使用 `AppContainer` 做轻量装配。对当前体量来说，维护成本更低，阅读门槛也更小。

### 3. 面向真实后端做了兼容

由于现网响应和 Swagger 并不总是完全一致，项目在 DTO 映射层做了不少双兼容处理，例如：

- `avatar` 既兼容字符串，也兼容对象
- 用户主页既兼容 `data.info` 包装，也兼容直接平铺
- 徽章列表既兼容数组，也兼容 `{ count, list }`
- 问题是否有最佳回答同时兼容 `accepted` 与 `accepted_answer_id`

### 4. 适合继续扩展

后续如果要继续补齐网页端能力，当前结构比较适合继续演进的方向包括：

- 通知中心
- 更完整的评论线程能力
- 点踩 / 更多 reaction
- 草稿自动保存
- 更完整的测试体系
- 引入 DI / CI / 静态检查
